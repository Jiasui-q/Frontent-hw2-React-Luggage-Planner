Collaboration policy

Programming assignments should be your own. You are not allowed to copy other students code, or copy/modify large chunks of code or entire projects found online (e.g. on GitHub).

You are allowed to consult online resources, e.g. Stack Overflow, to get answers to specific implementation questions.

LLM-based tools like ChatGPT and GitHub Copilot are helpful and increasingly common in industry, and you may use them to help you complete assignments, with the following caveats:

  • You should stick to using AI for line- or function-level edits; do not paste the entire chunks of the assignment description into ChatGPT (it’s pretty obvious when you do this). Programming assignments are not quite like real-world software development, because we often provide detail in the specification that would not normally be written out for you in a real-world project. If you use these tools, you should practice the skill of communicating with the tools rather than letting the assignment document do this for you.
  • We generally describe what we want you to learn from a particular exercise. Work through each task with the intention of practicing the skills we mention; if AI helped you get unstuck, do your best to come away with an understanding of the concepts targeted by the assignment.
  • When we ask you to think about the design and aesthetics of your application, we’re looking for your creative spark. AI can be great for helping you find the right CSS rules to style an element, but you should be making your own high-level decisions.
  • The goal of these assignments is to develop your skills to work independently on code implementations. LLMs can be part of this process, but when you prompt AI tools to help you write assignment code, you should not share the exact prompts you used with other students. This has a similar effect as sharing code, since it prevents other students from developing their own skills with the tools they are using. Of course, it's always okay to exchange high-level advice with your peers.

Part 1 · Reusable components

There are two big reasons to use components in React. The first is to break up your application’s code into a hierarchy; we did this in the Sidebar component, to avoid having to clutter the App.jsx file with all of the sidebar’s content.

The other is to create reusable UI elements so that you don’t need to rewrite the same HTML structure over and over.

Peek into src/luggage-overview/LuggageOverview.jsx. This is the code for the landing page of the application; it describes the different sizes of luggage bag that users are allowed to bring onto a flight. This page isn’t dynamic, and we won’t add any interactivity to it for now; it’s just informational.

It’s not very beautiful! You should think about how we might structure and style the markup so that it looks better; maybe the icon could be bigger or in a different spot. Maybe we can render the maximum weight in a way that stands out better. Make some design choices! In particular, you should choose a design that renders each luggage type more or less the same way (but with different content); the motivation for this will come up in a moment.

If you change the markup, you’ll notice that you have to change it in more than one spot to keep the luggage types consistent. Rather than repeating the same markup over and over, let’s extract this into a reusable React component!

Make a new function (either in the same LuggageOverview.jsx file, or in a new file that you import) that renders a single luggage type. The function should be a React component, meaning:

  • Its name should start with a capital letter
  • It should take in, as its first argument, the props of the component
  • It should return the JSX markup representing the HTML structure we want to render

Then, replace the individual luggage type sections in LuggageOverview.jsx with instances of your new component. It might look something like this: 

<LuggageType
  name={LUGGAGE_TYPES[0].name}
  ...
/>
<LuggageType
  name={LUGGAGE_TYPES[1].name}
  ...
/>

Now, you’re free to customize the structure and style of the component, and every change you make will affect both instances at once! Play around with the design until you’re happy with it.

Rendering lists

Let’s reap the benefits of having a reusable component. Add a new luggage type to the LUGGAGE_TYPES array (maybe we‘ll make it the first element of the new array, to keep it in order of weight): 

{
  icon: "🎒",
  name: "Personal item",
  maxWeightLbs: 15,
  description: "A small bag that fits under the seat in front of you.",
}

If you’d like, you can add a new copy of the component you made earlier to the page, to see if this worked. But rather than manually hard-coding every component, it’d be better to iterate over the LUGGAGE_TYPES array, automatically making a new component instance for each luggage type. Then, future updates to the array would automatically be reflected in the UI.

Render the list of luggage types dynamically, using the full LUGGAGE_TYPES array. The React docs show you how to do this in detail, but the simple principle is: include a component in the markup for each element of the array, and don’t forget the key prop! You don’t need to use a list HTML element (ul or ol); the docs show this as an example, but you can render arbitrary dynamic element trees using this approach.

The result is that your main component rendering code won’t need to change at all if you want to add, remove, or reorder luggage types; only the LUGGAGE_TYPES array needs to be updated.

Part 2 · Adding interactivity

Not all luggage is allowed for every plane ticket. In our airline’s “Basic Economy” fare class, only a personal item and a carry-on bag are allowed.

Let’s add an option to the page that lets the user select whether they are on a Basic Economy fare, and if so, do not show the option for checked luggage.

We’ll start out with a simple design for this feature, a button that, when clicked, toggles whether the user is on a Basic Economy fare or a normal (full-featured) fare.

In HTML, we can use the <button> tag to make a clickable, interactable button: 

<button>Toggle Basic Economy</button>

If you add this to your page, it won’t do anything yet, but it’ll look something like this:

Now, how do we make it so that clicking this button can control the displayed luggage types?

In React, we can attach program state to a component, ensuring that the state value is remembered even as the component re-renders. We do this with the useState function, which we’re already using in App.jsx to decide which page to render.

Have a look at React’s documentation about useState, especially the examples!

With useState, add a Boolean state variable (basicEconomy) to the LuggageOverview component. Don’t forget to import the useState function (you can see how this is done in App.jsx, or see if you can get your IDE to help you import the function automatically).

Add a button, too, so that when the user clicks the button, the basicEconomy state variable toggles: if it was true, it becomes false, and vice versa. In React, you can add a click handler to a button using the onClick prop; the examples in the documentation do this a lot.

Then, use the value of the basicEconomy state variable to decide whether to render the checked luggage option.

In the last part, we made a reusable component to render a luggage type, and then rendered the list so that we don’t have to write separate code for each luggage type. This is a good pattern, but now we need a way to distinguish the checked luggage type from the others. You might do this by adding a new field to the luggage type objects in the LUGGAGE_TYPES array, something like allowedInBasicEconomy: 

[..., {
  ...
  name: "Carry-on",
  allowedInBasicEconomy: true,
  ...
},
{
  ...
  name: "Checked bag",
  allowedInBasicEconomy: false,
  ...
}]

Then, based on the state variable, you can choose whether or not to render luggage types that have allowedInBasicEconomy set to false.

You can do this either by filtering the array (so that you only render the relevant components in the first place) or by returning null (rather than HTML markup) from components that shouldn’t appear on the page.

Check that this button works correctly before moving on:

What’s the problem here?

The state isn‘t visible to the user! Although the interactivity works, it’s not clear whether Basic Economy is currently active or not.

One thing we could do is actively render the state into the page:

let buttonText = "Toggle Basic Economy";
if (basicEconomy) {
  buttonText += ' (currently ON)';
} else {
  buttonText += ' (currently OFF)';
}

...

<button onClick={...}>
  {buttonText}
</button>

This works!

But we’re reinventing the wheel a bit here... aren’t there already well-designed togglable components that communicate their current state?

Controlled components

This really ought to be a checkbox. HTML offers a checkbox element, a variant of the <input /> tag: 

<label>Basic economy: <input type="checkbox" /></label>

(including a label ensures that the browser knows that this text is associated with the form control, which is great for accessibility; it will also make it so that clicking on the text toggles the checkbox. See below:)

If you add this to your page, you’ll see the checkbox. It will even be interactive, without you needing to write any code! But it won’t control (or reflect the value of) our state variable.

In React, we control <input /> elements the same way we just controlled our button, by rendering the current state and then accepting interactions that update the state.

The checkbox component takes a checked prop, which decides whether the checkbox shows as being checked. If you set this to true or false, the checkbox will actually stop being interactive (since React is forcing its value to whatever the checked prop is set to).

To make the checkbox interactive again, we need to use the state variable as the value for the checked prop. Then, we should update our state variable (basicEconomy) when the box is checked or unchecked. We can do this with an onChange handler, as shown here in the React docs.

Even if we change the state variable for some other reason (say, the user chose some other option that is incompatible with Basic Economy, so we set the state variable to false ), this technique will update the checkbox whenever the state changes. The change doesn’t only need to come from the user clicking the box.

A component that works like this — routing state through a state variable — is called a controlled component, since React is in charge of what state to render, as well as updating it when the user interacts with the page.

Add this working checkbox to your page, and remove the button you made earlier.

You’re encouraged to try a more interesting design than a checkbox in your final submission — a drop-down box? a set of tabs? a toggle switch? — but the checkbox design (as long as it’s clear and aesthetically appealing!) is acceptable here.

Part 3 · Luggage planning

Let’s create a more interesting interactive page, one that provides some value to the user. On the second page (the “Planner”), we’ll give the user the ability to plan out what’s going to go in each of two bags: a personal item, and a carry-on.

To keep it simple, we’ll hard-code the application so that it assumes the user just has one of each of these bags.

Implement this application in Planner.jsx. It should have the following features and implementation details:

  • Users can add items to either bag, specifying the name and weight (in pounds) of items as they are added.
    • Use text fields for data entry (not window.prompt, which is a convenient but limited way to get user input)
    • Prevent users from adding items (e.g., by disabling the button(s)) if the “name” field is empty or the weight is negative
    • After successfully adding an item, clear the input fields
  • Show users the items they have added so far in each bag
    • Use a React component to render each item, like we did in Part 1
  • Let users delete items they have added
  • Calculate and show the total weight of each bag
  • Show a warning if the weight of a bag exceeds its limit
    • You can hard-code a limit of 15 lb for the personal item, and 22 lb for the carry-on

You’re welcome to add additional features or interactions as you see fit! Make your application beautiful, too! If you add anything especially fancy, please explain it with instructions in the page so we know how to use your application.

Here are some tips we think will help you implement this part:

Controlled text fields

You can use text fields to get user input. The HTML tag for a text field is another variant of the <input /> tag: 

<input type="text" />

You can also use type="number" to get a numeric input field! As with checkboxes, we need to control these inputs so that we can use their values.

Arrays in state variables

Placing arrays in state can be trickier than simpler values like Booleans, numbers, and strings. This is because, to call the state update function (the second value returned by useState), you need to provide a new array representing the changed value.

It’s important not to change the existing state array directly (e.g., by using the push function). React expects state values to be immutable, so you can only update the state value by creating a new array with your requested changes and replacing the old array entirely.

There’s lots of detail on how to do this in the React documentation!

Making a delete button

You might find that you have some trouble making a “Delete” button within a component. When it’s clicked, the button should update the state array in the parent component, removing that item from the state array.

You don’t have direct access to the parent’s state in a child. Instead, in the code for the Planner component, you’ll need to provide the Item component (or whatever you call it) with a function that can be used to delete that item.

This page of the React documentation offers a lot of detail, through a worked example.

Disabling buttons

You can disable a button by adding the disabled prop to it. If you set this prop to true, the button will be disabled; if you set it to false, it will be enabled. Use a JavaScript expression (in {curly braces}) to decide, based on the program state, whether the button should be enabled or disabled.

Part 4 · Adding a new page

Finally, edit App.jsx and Sidebar.jsx to add a new page to the application. This page should be called “Summary” or something similar.

The Summary page is only required to contain one piece of information: the total weight of all items added in the Planner application, across all bags. You are welcome to show additional insights to the user!

If you added one or more state variables in the Planner to track the items in each bag, you’ll notice that navigating to another page clears the state entirely, and you certainly won’t have access to that state variable in the Summary page. Both Planner and Summary are children of the App component, but they can't directly share state between one another.

How do we fix this?

The answer is to lift state up: put the luggage state variable into the App component instead, and give access to it in both Planner and Summary. The React documentation offers some guidance. You’ll need to make sure you can still update the program state from the Planner page! This will likely involve passing a function into the props of <Planner>.

If this is starting to feel like a lot of props being passed back and forth, consider using Context to manage how data is passed to components. The useContext page provides great exercises on how to update context within your application!

Using React Context here is optional, but if you choose to use it, you can earn some credit for it in Part 5 of this assignment.

Checklist

  • A new page (Summary) is linked in the sidebar.
  • Navigating between pages does not lose information about which items have been added to bags.
  • The Summary page includes the total weight of all items in all bags, updating as the user updates the contents of the bags.

Grading for this assignment, scored out of 40 points, includes 30 points for functionality.

To earn the remaining 10 points, you must improve the aesthetics of your application. You can also earn points by implementing extra features.

New features

Here’s a list of features we will award points for:

1 point each
